Pitfalls & Tuning

Things that silently break LCStool runs — and how to diagnose them.

1. Environment & compatibility

MATLAB version. LCStool v1.0 was written against MATLAB R2013a. Features that regularly break on newer releases:
  • griddedInterpolant boundary behaviour changed after R2017b — NaN handling of the 'linear' extrapolation mode is now stricter. Use 'none' explicitly if you want out-of-domain queries to return NaN (the toolbox assumes so).
  • parfor requires Parallel Computing Toolbox. Without it, compute_cauchy_green and integrate_line still run, but serially and ~4–8× slower.
  • ode45 tolerances default to 1e-3/1e-6; all demos explicitly set AbsTol = RelTol = 1e-6. Leaving the defaults produces visible "wobble" in shrink-line integration.

2. The axis-order trap

This is the single most common bug in user code.

MATLAB's ndgrid and meshgrid use opposite axis conventions. LCStool stores its grids as [x₁ y₁ … xₙ yₙ] pairs in a column vector, built from ndgrid. When you later call reshape to turn a scalar field into an image, the resolution vector must be flipped:

ftle_matrix = reshape(ftle_vec, fliplr(resolution));  % correct
ftle_matrix = reshape(ftle_vec, resolution);          % WRONG — transposes result

Why: resolution = [nx, ny] but reshape fills column-major, so the leading dimension must be the fast one (y). All LCStool helpers already do this internally; you only need to remember it when you pull raw vectors out of cgStrain.

Sanity check

% After compute_cauchy_green returns
imagesc(cgEigenvalue(:,:,2));   % should look like a proper flow field
% If it looks "sheared" or axis-rotated, your reshape is wrong.

3. NaNs in the eigen-fields

You open imagesc(cgEigenvalue(:,:,2)) and see white patches. Ranked by frequency:

  1. Trajectories leaving the domain. When a particle integrated by compute_cauchy_green exits the region where flow.derivative is defined, the interpolant returns NaN, and ∇F — and therefore λ₂ — becomes NaN. Fix: enlarge the domain, shorten timespan, or pad the velocity data with zeros before building the interpolant (see ocean_dataset.m).
  2. Auxiliary grid too large. With method = 'finiteDifference', each grid point spawns a 4-point stencil offset by auxGridRelDelta. If that offset pushes the stencil outside the domain, the corner trajectory fails. Reduce auxGridRelDelta from the default 0.01 to e.g. 0.005.
  3. Non-finite velocity. Velocity fields with NaN values anywhere (common in AVISO ocean data near coastlines) contaminate every trajectory that touches them. Pre-process: replace NaN with 0 (for closed coasts) or mask them out and restrict the analysis domain.
  4. Stiff ODE. Very rarely, ode45 gives up on a stiff point and returns NaN. Switch to ode23 or ode15s (pass via odeset and a custom odeSolverOptions).
Quick NaN map: 
figure, imagesc(isnan(cgEigenvalue(:,:,2)))
title('NaN mask — white = bad trajectories')

4. Poincaré section placement

The elliptic-LCS detector scans a Poincaré section — a user-supplied line segment — and looks for λ-lines that cross it and return to themselves. Section choice is critical:

If the section…ResultRemedy
misses the vortex coreNo closed orbits foundPlace the section to cut through the vortex — e.g. along the symmetry axis.
passes through a hyperbolic pointReturn map has a singularity; bisection fails near itOffset the section by ≈5% of the vortex radius.
is too shortOutermost orbit is cut offExtend until the return-distance plot shows a clear rise at both ends.
is too longSpurious zeros from adjacent structuresShorten or split into multiple sections; use showPlot.orbitsPoincareSection = true to debug.

Inspecting the return-distance map

% Inside poincare_closed_orbit.m workflow
showPlot.poincareSection = true;
showPlot.returnDistance  = true;       % 1-D plot of d(s) along section
% A "good" section shows d(s) crossing zero once per closed orbit, with
% changes of sign well-separated. Multiple rapid oscillations = bad section.

5. λ-sweep granularity

Elliptic LCS are stacks of closed λ-lines for lambda ∈ [lambdaMin, lambdaMax]. Increasing the number of λ values:

  • Finds thinner Lagrangian vortices (the outermost orbit survives over a narrow λ window).
  • Linearly increases cost — each λ is a full Poincaré scan.
Typical values.
  • Double Gyre: lambda = 0.9 : 0.01 : 1.1 (21 values) — plenty.
  • Bickley Jet: 0.93 : 0.005 : 1.07 — core vortices are slim; finer step.
  • Ocean data: start with 0.8 : 0.02 : 1.2, refine around where orbits appear.
Do not include λ = 1 exactly if you plan to compare to Haller's stretching-rotation vortices — at λ=1 the η-field degenerates to a ξ₂ line, which is the shear skeleton, not a vortex boundary.

6. Orientation discontinuities

The Cauchy-Green eigenvectors (ξ₁, ξ₂) are line fields, not vector fields: their sign is arbitrary. LCStool integrates them using a valueHandle that flips sign when the local orientation disagrees with the previous step by more than a threshold angle (orientDiscontAngle, default π/2).

Warning "large angle discontinuity". Printed by integrate_line when a sign flip of more than orientDiscontAngle is required. This usually indicates:
  • The eigenvector field is genuinely singular at that point (λ₁ = λ₂ — an umbilical point). Safe to ignore if isolated.
  • The auxiliary grid is too coarse to resolve a sharp rotation. Decrease auxGridRelDelta.
  • The ODE step is jumping across a discontinuity line. Tighten ODE tolerances (RelTol = 1e-8).
Use plot_orient_discont to overlay the discontinuity map on your FTLE plot and see if the warning tracks a real feature.

When the run is too slow

  1. Profile: profile on; poincare_closed_orbit(...); profile viewer
  2. Typical hot spot is the interpolant inside flow.derivative — make sure you pre-build griddedInterpolant once and capture it by closure, not rebuild on every call.
  3. For elliptic LCS on a fine grid, most time is spent in integrate_line — this is where parfor pays off.

8. Quick sanity checks

Run these after any new flow field is plugged in: 
% 1. Flow map is conservative to ~1e-12 (incompressible test)
lam1 = cgEigenvalue(:,:,1);
lam2 = cgEigenvalue(:,:,2);
disp(max(abs(lam1(:).*lam2(:) - 1)))     % should be <~1e-10

% 2. FTLE is non-negative
ftle = 1/(2*lcs_T) * log(lam2);
disp(min(ftle(:)))                       % ≥ 0 up to rounding

% 3. Eigenvectors are unit length
xi1 = cgEigenvector(:,:,1:2);
disp(max(abs(sum(xi1.^2,3) - 1)))        % <~1e-12
If any of these fails by more than rounding, suspect NaNs or an axis-order bug.

9. Reproducibility notes

  • ode45 is deterministic for a fixed version of MATLAB, but results can shift in the 5th decimal across MATLAB releases. Always version-pin for papers.
  • parfor loop order is non-deterministic but the result is deterministic, because each iteration writes a unique slice. Numerical equality is safe to assert.
  • When comparing against published figures from Onu, Huhn & Haller (2015), the demos ship exact parameter values; changing timespan or resolution will produce subtly different-looking LCS.

10. Known issues in v1.0

  • animate_flow.m uses getframe, which on headless Linux produces blank frames. Work around with set(gcf,'Visible','off') and print to file per step.
  • If your grid has only one point along an axis (1D test), reshape collapses and later plots fail. The toolbox is strictly 2D.
  • The valueHandle class is a legacy workaround for MATLAB's lack of object references pre-R2008a. It works fine but produces hard-to-read stack traces — when debugging, set a breakpoint inside integrate_line_derivative, not in the ODE callback.
