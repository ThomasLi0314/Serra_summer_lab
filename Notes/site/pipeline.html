Pipeline

The five stages of an LCS computation, end to end.

Overview diagram

Every demo follows the same five stages. The driver script only changes parameters — the computational skeleton is identical.

Stage 1 — Flow definition

You describe a flow to LCStool with four things:

NameTypeMeaning
domain$2\times 2$ double[xMin,xMax; yMin,yMax]
timespan$1\times 2$ double[t0, tF]
resolution$1\times 2$ integer[Nx, Ny]
lDerivativefunction handleRHS of $\dot{\mathbf{x}} = \mathbf{v}(t,\mathbf{x})$

The derivative must follow a strict column-vector convention (explained in How-To Guide → conventions):

$$ \mathbf{x}_{\text{in}} = [x_1, y_1, x_2, y_2, \ldots, x_N, y_N]^\top \ \mapsto\ \mathbf{v}_{\text{out}} = [u_1, v_1, u_2, v_2, \ldots, u_N, v_N]^\top. $$

Why column-stacked? Because ode45 accepts only a single state vector, and LCStool integrates many particles simultaneously in a single coupled ODE call (see integrate_flow's coupledIntegration mode).

A minimal wrapper for a flow with parameters p1, p2:

lDerivative = @(t, x, ~) derivative(t, x, false, p1, p2);

The trailing ~ in the handle is the useEoV slot (equation of variation). Always pass false unless you explicitly want the EoV machinery.

Stage 2 — Cauchy-Green & eigen-decomposition

This single call does the heavy lifting:

[cgV, cgD] = eig_cgStrain(lDerivative, domain, resolution, timespan, ...
                          'incompressible', incompressible, ...
                          'odeSolverOptions', cgStrainOdeSolverOptions);

What happens inside

  1. initialize_ic_grid builds the main grid — a flattened Nx·Ny × 2 list of initial conditions.
  2. An auxiliary grid is built around each main grid point: four extra seeds at $(\pm\delta, 0)$ and $(0, \pm\delta)$ from every main point, with $\delta = \text{auxGridRelDelta} \cdot \Delta x$ (default $10^{-2}$).
  3. All these seeds are advected through $[t_0, t_F]$ with ode45, coupled into a single ODE state for speed. Each "particle" is just two coupled equations, so the full state has length $2 N$.
  4. The final positions give $\nabla F$ via central differences (see Math §2).
  5. From $\nabla F$, the three independent entries of $C = \nabla F^\top \nabla F$ are assembled and diagonalised at every grid point.
  6. If 'incompressible', true is set, $\lambda_1 \gets 1/\lambda_2$ everywhere that $\lambda_2 > 1$; other points become NaN.

Data layout of the output

VariableShapeContent
cgD$N \times 2$$[\lambda_1,\lambda_2]$ at every main-grid point ($\lambda_1 \le \lambda_2$)
cgV$N \times 4$$[\xi_{1x}, \xi_{1y}, \xi_{2x}, \xi_{2y}]$ packed column-wise

Indexing convention: row-$k$ of both arrays corresponds to the $k$-th row of the flattened meshgrid (y changes fastest; reshape(…, fliplr(resolution)) recovers the 2-D layout).

Stage 3 — Hyperbolic LCS (shrink & stretch lines)

Two calls to the same function, differing only in which eigenvalue/eigenvector pair is supplied:

% Repelling (shrink) lines — follow ξ1, seed at maxima of λ2
shrinkLine = seed_curves_from_lambda_max( ...
    shrinkLineLocalMaxDistance, shrinkLineMaxLength, ...
    cgD(:,2), cgV(:,1:2), domain, resolution, ...
    'odeSolverOptions', shrinkLineOdeSolverOptions);

% Attracting (stretch) lines — follow ξ2, seed at minima of λ1
stretchLine = seed_curves_from_lambda_max( ...
    stretchLineLocalMaxDistance, stretchLineMaxLength, ...
    -cgD(:,1), cgV(:,3:4), domain, resolution, ...
    'odeSolverOptions', stretchLineOdeSolverOptions);

Note the sign flip on cgD(:,1): the function always finds local maxima of its input, so to get minima of $\lambda_1$ you pass $-\lambda_1$.

Algorithm walk-through (per seed)

  1. Find all local maxima of the input scalar with a distance threshold (circular-mask neighbourhood). Sort descending.
  2. Take the largest unused maximum as a seed.
  3. Integrate the eigenvector field forward with integrate_line (sign-flip logic keeps orientation continuous) until:
    • arc length exceeds curveMaxLength, or
    • the trajectory leaves domain, or
    • it hits NaN (e.g., a point where eigenvectors are ill-defined).
  4. Integrate backward from the same seed in the same way.
  5. Concatenate the two halves, mark the grid cells they cross as "used" (circular mask) so subsequent seeds are spaced at least distance apart.
  6. Repeat from step 2 until no unused maxima remain or nMaxCurves curves are produced.
Why the distance threshold matters. Without it, every local maximum would spawn a curve and they would overlap into an indecipherable mess. A typical setting is 2·deltaX for shrink lines and 10·deltaX for stretch lines — stretch lines can share seeds more readily.

Post-processing: remove interior strainlines

If elliptic LCSs have been computed, strainlines that lie inside a vortex boundary are removed by remove_strain_in_elliptic. This makes the visualisation cleaner (the vortex interior is not a hyperbolic region).

Stage 4 — Elliptic LCS (λ-lines & Poincaré sections)

Three layered function calls build this:

[clPos, clNeg] = poincare_closed_orbit_range( ...
    domain, resolution, cgV, cgD, lambda, poincareSection, ...
    'forceEtaComplexNaN', true, ...
    'odeSolverOptions', lambdaLineOdeSolverOptions);

ellipticLcs = elliptic_lcs(clPos);
ellipticLcs = [ellipticLcs, elliptic_lcs(clNeg)];

Nesting structure

poincare_closed_orbit_range (loops over λ values) └─ lambda_line (build η+, η- for this λ) └─ poincare_closed_orbit_multi (loops over Poincaré sections) └─ poincare_closed_orbit (one section, one η-field) └─ integrate_line (single orbit integration)

Per-section logic inside poincare_closed_orbit

  1. Seed numPoints particles uniformly along the section.
  2. Integrate each with integrate_line; stop when the trajectory returns to the section (event function ode_events_poincare).
  3. Rotate into a frame where the section lies on the x-axis; the y-coordinate of each return point is the return distance $d$.
  4. Find zero-crossings of $d(s)$ vs. parametric coordinate $s$ along the section.
  5. For each zero-crossing, run a bisection refinement (nBisection iterations) on the two neighbouring seeds to locate the root precisely.
  6. Discard roots whose neighbour distances exceed dThresh × sectionLength (spurious, non-smooth zero-crossings).
  7. Integrate each surviving root as a closed orbit; sort innermost → outermost by distance from the section's start.

The λ sweep

A single $\lambda$ often yields several nested closed orbits. The outer loop poincare_closed_orbit_range sweeps a range like lambda = 0.9:0.02:1.1; each $(\lambda, \Sigma)$ pair produces an innermost + outermost orbit (or NaN).

Finally, elliptic_lcs picks, per Poincaré section, the outermost orbit with the largest enclosed area across the whole λ-sweep. That is the declared elliptic LCS.

Stage 5 — Plotting & analysis

All plotting helpers follow a common idiom:

  1. Call setup_figure(domain) to get an axes with equal aspect ratio and the correct limits.
  2. Call the specific plotter: plot_ftle, plot_elliptic_lcs, etc.
  3. Customise handles returned by the plotters (e.g., set colour or linewidth).

The canonical LCS figure in every demo is produced by:

hAx = setup_figure(domain);
title(hAx, 'Repelling and elliptic LCSs');

% Elliptic in green, plotted first so it sits on top via uistack later
hEll = plot_elliptic_lcs(hAx, ellipticLcs);
set(hEll, 'color', [0 .6 0], 'linewidth', 2);

% Repelling in red
hRep = cellfun(@(p) plot(hAx, p(:,1), p(:,2)), shrinkLine, 'UniformOutput', false);
hRep = [hRep{:}];
set(hRep, 'color', 'r');

uistack(hEll, 'top');

See the dedicated Plotting chapter for each helper's arguments.
